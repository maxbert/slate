    Introduction

    Wellcome to the PEACH API. This is a API for storing, querying and analyzing patient reading data stored on the PEACH cluster.

    In order to use this API, you must have your data in the basic PEACH format. The PEACH format relies on two tables, one of patients and one of readings.

    You can read more about how to format your data in the Importing Data section of this Documentation.

    Currently, there are no native API implementations, however, we have included curl examples of querying the http API in this Documentation.

    Authentication

    Authentication is based on JWT tokens. Contact your organisation if you are unsure of what your token is. For each request, you are required to send an authorisation header with the bearer parameter.

    Authorisation looks like this:

    # With shell, you can just pass the correct header with each request
curl "api_endpoint_here"
  -H "Authorization: Bearer 123456789"

    Make sure to replace 123456789 with your API key.

    Authorization: Bearer 123456789

    You must replace 123456789 with your personal API key.

    Patients

    Each patient object is composed of a Unique_ID and then patient parameters specific to your organisation. These parameters can be any value, Booleans are represented by 0 and 1. These paramters are used to sort patients, so it is advisable (though not required) to make sure that all patients have the same paramters in their patient object.

The parameter keys should be consistent over all readings. Readings must be unique over date and patient_id (two readings cannot have the same date and patient_id).

    This endpoint retrieves all readings within the given time range, where time is relative to the latest entered reading.

    This system works by calculating a days before for each reading, where days before is the given readings date subtracted from the most recent reading available of the given patients data.

    For example, if patient 10 has 3 readings

    ID Date
    1 12/12/1999
    2 13/12/1999
    3 14/12/1999

    Then those readings would get days before values like this

    ID Date Days Before
    1 12/12/1999 2
    2 13/12/1999 1
    3 14/12/1999 0

    Thus, a query like GET http://51.140.9.200/readings?start=1&end=0 would return readings 2 and 3 but not 1

    This allows for multiple patients to be compared on a time scale, even if they did not start collecting readings on the same date.

    HTTP Request

    GET http://51.140.9.200/readings?start=1&end=0

    Query Parameters

    Parameter Default Description Required
    start 1000 number of days before the latest reading to query no
    end 0 number of days after the latest reading to query no

    Return Object

    Key Description
    patients An array of patient reading objects

    Patient Reading Object

    Key Description
    id A patients unique identifier.
    readings a list of reading objects

    Reading Object

    Key Description
    patient_id A patients unique identifier.
    date The date of the reading in YYYY-MM-DD
    unique_key A concatenation of the patient_id, date and organisation
    days_before The number of days between this reading and the given patietns latest reading.
    id the database identifier of the reading

    The reading object also includes keys and values specific to your organisation

    Remember to send your bearer token!

    This endpoint retrieves all readings categorized by a patient parameter.

    HTTP Request

    GET http://51.140.9.200/readings?column_name=YOUR_PATIENT_PARAMTER1&breakpoint=100

    Where YOUR_PATIENT_PARAMTER1 is replaced with a given patient parameter

    Query Parameters

    Parameter Default Description Required
    column_name null The patient paramter on which to divide the patient readings. This column name must be one returned by the /columns route yes
    breakpoint 1 number which to divide patients on. Greater than or equal to breakpoint will place a patients readings in yes_patients, less than breakpoint will place the patients readings in no_patients no

    Return Object

    Key Description
    yes_patients An array of patient reading objects where the column_name value is greater than or equal to the breakpoint
    no_patients An array of patient reading objects where the column_name value is less than the breakpiont

    Patient Reading Object

    Key Description
    id A patients unique identifier.
    readings a list of reading objects

    Reading Object

    Key Description
    patient_id A patients unique identifier.
    date The date of the reading in YYYY-MM-DD format
    unique_key A concatenation of the patient_id, date and organisation
    id the database identifier of the reading

    The reading object also includes keys and values specific to your organisation

    Remember to send your bearer token!

    This endpoint uploads a json list of readings into the database

    The readings must be separated by patient, with each patient having its own list of readings.

    Each reading must consist of one or more reading parameters, and a date, which should be a string in YYYY-MM-DD format

    Readings must be unique across patient_id and date, no two readings can exist for the same patient on the same date.

    Readings which already exist on the given date for a given patient will be skipped on upload

    HTTP Request

    POST http://51.140.9.200/readings

    Return Object

    Key Description
    message success or error message

    Errors

    The Peach API uses the following error codes:

    Error Code Meaning
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your API key is wrong.
    404 Not Found -- The specified resource could not be found.
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
